POODLE IPv4 Addresses with Trusted Certificates

This dataset contains websites on the public IPv4 address space that support SSL, but do not support any version of TLS. We urge these sites to deploy TLS. More information and statistics can be found at https://poodle.io.

Dataset

Our latest scan finished yesterday (Tuesday, October 14) at 12:30 PM EDT and can be downloaded at ipv4-sslv3only-trusted.2014-10-14.csv.gz. We will be posting updated data when new scans complete.

Scan Methodology

In order to generate a list of domains that only support SSLv3 (but not any version of TLS), we performed a scan of the public IPv4 address space using the ZMap Internet Scanner and then attempted to perform a TLS 1.2 handshake with each IP that responded to ZMap on port 443. The list here contains sites with browser trusted certificates (validated with Mozilla NSS root store) that did not support any version of TLS and downgraded the connection to SSLv3. Domain names were generated based on the CN (common name) on the presented certificate.

Because we are not connecting to specific domains using SNI (because there is authoritative list of websites on the Internet), there are several caveats to this dataset. First, websites that are only accessible through SNI may not appear on the list even if they only support SSLv3---because we have no way to connect to them. Second, if an organization presents the browser-trusted certificate for their main site on a secondary IP address, this will show up on our list---even if their main site supports TLS. Third, because we initiate a TLS 1.2 connection, it is possible that sites could support SSLv3, but become confused by our TLS 1.2 handshake and terminate the connection instead of negotiating down to SSLv3. This shouldn't happen, but there are some implementations that do not fully follow the specification.

Correction to Previous Dataset

Due to a parsing error, our original dataset incorrectly listed several domains (including google.com) as only supporting SSLv3. This dataset was removed after we discovered the bug and correct data has been reposted. We thank Adam Langley and Jacob Hoffman-Andrews for catching this error.
